# 翻译质量问题记录：短文本重复输出

## 问题概述

- 场景：工作台输入短中文（如 `测试测试`）并翻译到英文。
- 现象：返回大量重复 token，例如：
  - `Test test test test testing test test tests test test to test`

该问题会影响用户对翻译能力的信任，需要记录并跟踪处理。

---

## 复现条件

- 模型：本地内置 `zh-en`（CTranslate2 + sentencepiece）
- 输入：`测试测试`
- 目标语言：英文
- 运行方式：可通过 UI 复现，也可通过脚本直调接口复现。

---

## 实验对照（绕过 UI，直接模型推理）

对同一输入做了不同 decoding 参数对照，均出现重复问题：

1. 当前参数（beam=4 + repetition_penalty + no_repeat_ngram）  
输出：`Test test test test testing test test tests test test to test`

2. beam=1  
输出：几乎全量 `test` 重复（更差）

3. 缩短 max_decoding_length  
输出：仍重复，只是长度变短

---

## 结论

该问题**主要由模型本身在“超短文本/重复短语”场景下退化导致**，不是 UI 渲染问题。  
调整 decoding 参数只能部分缓解，无法根治。

---

## 影响范围

- 超短中文输入（2~6 字）风险最高
- 混合文本或语义不足输入也可能触发
- UI 与接口均会暴露该问题

---

## 建议方案

### 短期（工程兜底）

1. 增加翻译结果质量门控：
   - 检测明显重复模式（如高频重复词/重复 n-gram）
   - 命中后返回“结果质量不稳定”提示
2. 提供降级策略：
   - 对异常结果做裁剪/去重
   - 或直接提示用户补充上下文后重试

### 中期（模型替换）

1. 评估更稳的中英离线翻译模型（体积允许前提下）
2. 用真实业务短句集做离线评测（重复率/可读性/正确率）
3. 决策是否升级内置模型

### 交互优化

1. 输入过短时提示：建议输入完整句子以获得更稳定译文
2. 对“重新翻译”按钮增加质量提示文案
