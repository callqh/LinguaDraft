# 语言识别技术选型文档（桌面端离线写作翻译工具）

## 1. 背景

当前项目是一个基于 Electron 的桌面端离线写作翻译工具，核心流程包括：

- 键盘输入文本
- 语音录入文本
- 自动识别输入语种
- 根据目标语言执行翻译
- 历史记录支持重翻译

在“自动识别输入语种”这个能力上，需要解决两类问题：

1. **文本语言识别**  
   用户键盘输入文本后，系统判断当前文本是中文、英文、日文、韩文等哪种语言。

2. **语音语言识别**  
   用户语音输入后，系统判断当前语音属于哪种语言，并返回转写结果。

本次技术选型的重点是：

- 文本语言识别是否必须依赖本地模型
- 是否可以直接使用代码库
- 对不同方案做技术对比
- 给出最终推荐方案

---

## 2. 选型目标

本项目对语言识别方案的要求如下：

- 桌面端可用
- 尽量轻量
- 识别速度快
- 对中英场景效果稳定
- 可平滑支持后续多语言扩展
- 与本地翻译、本地语音识别架构兼容
- 工程接入成本低
- 不额外增加过重的模型管理负担

---

## 3. 需要区分的两个能力

## 3.1 文本语言识别

输入是纯文本，例如：

- `今天先整理需求`
- `We should finish the MVP first`
- `これはテストです`

目标是识别文本语言。

---

## 3.2 语音语言识别

输入是录音音频。

目标是识别：

- 用户说的是哪种语言
- 转写结果是什么

这一类能力通常已经包含在 ASR（语音识别）模型中。`faster-whisper` 是基于 CTranslate2 的 Whisper 实现，属于可本地运行的 ASR 方案。:contentReference[oaicite:0]{index=0}

---

## 4. 可选方案

本项目可考虑以下四类方案：

1. 纯规则 / 启发式判断
2. `franc`
3. `CLD3`
4. `fastText lid.176.ftz`

---

## 5. 方案对比

## 5.1 方案 A：纯规则 / 启发式判断

### 思路

通过字符集和文本特征做粗判断，例如：

- 包含大量汉字 → 中文
- 包含大量拉丁字母 → 英文或其他西文
- 包含日文假名 → 日文
- 包含韩文 Hangul → 韩文

### 优点

- 最轻量
- 不依赖外部模型
- 不依赖第三方推理库
- 实现成本最低
- 中英二选一时可作为快速兜底策略

### 缺点

- 不适合多语言精确判断
- 对法文、德文、西班牙文、意大利文等拉丁字母语言区分能力很弱
- 对短文本、混合文本不稳定
- 几乎无法输出可靠置信度
- 难以作为正式产品方案长期维护

### 适用场景

- demo
- 临时兜底
- “中文 / 非中文”粗判断
- 极简 MVP

### 结论

**不建议作为正式主方案。**  
可以保留为降级策略，但不建议作为项目主识别能力。

---

## 5.2 方案 B：`franc`

### 基本情况

`franc` 是一个 npm 包，用于文本语言检测，包描述为“Detect the language of text”，支持 **186 种语言**。:contentReference[oaicite:1]{index=1}

### 技术特点

- 直接作为 JavaScript / TypeScript 库使用
- 不需要单独下载模型文件
- 接入成本低
- 适合 Electron / React / Node 体系

### 优点

- 非常轻量
- 使用简单
- 不需要额外模型目录管理
- 适合文本输入场景
- 非常适合作为前端 / Node 侧的快速语种判断工具

### 缺点

- 本质上更适合文本场景，不适用于音频
- 对超短文本、混合文本场景可能不够稳定
- 无法统一到“本地模型管理体系”里
- 作为正式产品能力时，扩展性和可控性不如模型化方案

### 工程评估

对于当前项目，如果只想解决：

- 中文 / 英文 / 日文等文本识别
- 并且希望最低接入成本

那么 `franc` 是最容易落地的方案。:contentReference[oaicite:2]{index=2}

### 结论

**适合作为文本语言识别的轻量实现方案。**  
尤其适合第一阶段或中轻量正式版本。

---

## 5.3 方案 C：`CLD3`

### 基本情况

Google 的 `CLD3` 官方仓库明确说明：  
**CLD3 是一个神经网络语言识别模型，这个包包含推理代码和一个训练好的模型。** :contentReference[oaicite:3]{index=3}

### 技术特点

- 看起来像“一个库”
- 但本质上仍然是“库内嵌模型”
- 输入文本后，提取字符 n-gram 特征并由神经网络进行识别。:contentReference[oaicite:4]{index=4}

### 优点

- 识别能力通常比纯规则更强
- 属于成熟的语言识别方案
- 不需要单独管理 `.bin` / `.ftz` 模型文件

### 缺点

- 本质仍然是模型推理，只是模型被库打包在里面
- 在 Electron / Node 环境中接入和跨平台构建，通常比 `franc` 更麻烦
- 与本项目的 Python sidecar / CTranslate2 主线不完全统一
- 社区实际反馈中，对复杂场景并非总是稳定；例如 Mastodon 曾公开提到他们使用过 `cld3`，但效果并不理想。:contentReference[oaicite:5]{index=5}

### 工程评估

如果目标是：

- 想比 `franc` 更模型化
- 又不想单独维护模型文件

那 `CLD3` 可以考虑。  
但对当前项目而言，它的工程收益并不明显优于 `franc`，而复杂度却更高。

### 结论

**不建议作为本项目首选方案。**  
它比 `franc` 重，但又没有好到必须采用。

---

## 5.4 方案 D：`fastText lid.176.ftz`

### 基本情况

fastText 官方提供语言识别模型，可识别 **176 种语言**，并提供两个版本：

- `lid.176.bin`
- `lid.176.ftz`（压缩版）:contentReference[oaicite:6]{index=6}

其中压缩版 `lid.176.ftz` 大小约 **917 KB**。:contentReference[oaicite:7]{index=7}

### 技术特点

- 是典型的“代码库 + 外部模型文件”方案
- 需要显式加载模型文件
- 非常适合离线和本地模型管理体系

### 优点

- 模型非常小
- 多语言覆盖广
- 离线能力强
- 很适合纳入“统一模型管理体系”
- 可以放进 sidecar，与 ASR / 翻译模型统一管理

### 缺点

- 比 `franc` 多一层模型文件管理
- 接入复杂度高于纯 JS 库
- 对当前“只做中英文本识别”的需求来说，可能略显重

### 工程评估

如果项目已经决定采用本地模型体系，并且准备使用 sidecar 统一承载：

- Whisper ASR
- 翻译模型
- 文本语言识别

那么 `fastText` 是非常合理的正式方案。:contentReference[oaicite:8]{index=8}

### 结论

**适合作为正式长期方案。**  
尤其适合“要统一纳入本地模型架构”的版本。

---

## 6. 语音语言识别方案对比

对于语音输入，不建议单独再找一个“纯代码语音语言识别库”。

### 推荐做法

直接由 ASR 模型返回语言结果。

`faster-whisper` 是 Whisper 的 CTranslate2 实现，强调：

- 相同准确率下可更快
- 内存占用更低
- 支持 8-bit 量化:contentReference[oaicite:9]{index=9}

因此在语音场景中，推荐直接使用：

- `faster-whisper-base`
- 由 ASR 返回：
  - 转写文本
  - 语言结果

### 优点

- 不需要额外维护单独的语音语言识别模块
- 链路最短
- 与本地 ASR 方案天然一致
- 工程复杂度最低

### 结论

**语音语言识别不单独选型，直接复用 ASR 模型返回结果。**

## 8. 最终推荐

## 8.1 推荐结论

### 文本语言识别

**推荐采用：`franc`（当前阶段）**

### 语音语言识别

**推荐采用：直接复用 `faster-whisper` 返回的语言结果**

---

## 8.2 推荐原因

### 原因 1：最符合当前项目阶段

当前项目是从 MVP 往可用版本演进，已经有以下较重的本地模型能力要接入：

- `faster-whisper-base`
- `opus-mt-zh-en`
- `opus-mt-en-zh`

如果此时文本语言识别也上独立模型体系，虽然可行，但会额外增加：

- 模型清单管理
- 模型文件打包
- 模型初始化
- sidecar 接口复杂度

而 `franc` 作为纯 JS 库，接入成本最低，最适合作为当前阶段的文本语种识别方案。:contentReference[oaicite:10]{index=10}

---

### 原因 2：语音场景不需要额外选型

语音输入本身已经要接 `faster-whisper`。  
该方案本身就是本地多语言 ASR 路线，因此语音语种识别直接复用其输出即可，无需重复建设。:contentReference[oaicite:11]{index=11}

---

### 原因 3：控制系统复杂度

项目当前主线应该优先保障：

- 工作台主流程可用
- 本地翻译可用
- 本地语音识别可用
- 模型下载和模型管理可用

在这个阶段，把文本语言识别做成最轻实现，整体系统更稳。

---

### 原因 4：后续仍可平滑升级

如果后续出现以下情况：

- `franc` 对短文本识别不稳定
- 混合文本误判较多
- 需要更统一的本地模型管理体系

可以再升级为：

**`fastText lid.176.ftz`**

该模型体积很小，适合作为长期正式方案。:contentReference[oaicite:12]{index=12}

---

## 9. 推荐落地策略

## 第一阶段（当前推荐）

### 文本语言识别

- 使用 `franc`
- 在 renderer / Node service 层直接调用
- 不进入模型管理页
- 不需要单独下载

### 语音语言识别

- 使用 `faster-whisper` 返回语言结果
- 由 sidecar 返回：
  - `text`
  - `language`

---

## 第二阶段（可选升级）

若后续需要统一到完整本地模型体系：

### 文本语言识别升级为

- `fastText lid.176.ftz`

### 升级收益

- 与 ASR / 翻译模型统一纳入模型清单
- 能走统一的 sidecar 推理服务
- 更适合长期离线正式版本

---

## 10. 最终推荐方案（项目决策）

### 采用方案

- **文本语言识别：`franc`**
- **语音语言识别：`faster-whisper` 返回语言**
- **暂不采用 `CLD3`**
- **暂不采用 `fastText` 作为当前阶段默认实现**

---

## 11. 决策理由摘要

1. `franc` 是最轻、最易接入的文本语言识别方案，支持 186 种语言，适合当前项目阶段。:contentReference[oaicite:13]{index=13}
2. `CLD3` 虽然是神经网络语言识别方案，但本质是内嵌模型库，工程复杂度更高，当前阶段收益不明显。:contentReference[oaicite:14]{index=14}
3. `fastText lid.176.ftz` 很适合作为长期正式方案，但当前会增加一层模型管理复杂度，建议作为后续升级路线。:contentReference[oaicite:15]{index=15}
4. 语音语言识别直接复用 `faster-whisper` 输出，是最合理的架构选择。:contentReference[oaicite:16]{index=16}

---

## 12. 给 Codex 的实施要求

请按以下决策落地：

1. 文本语言识别采用 `franc`
2. 语音语言识别不单独实现，直接使用 `faster-whisper` 返回语言结果
3. 文本语言识别先不纳入本地模型清单管理
4. 预留后续将文本语言识别替换为 `fastText lid.176.ftz` 的接口层
5. 服务层接口保持统一，例如：

```ts
interface LanguageDetectService {
  detectTextLanguage(text: string): Promise<{
    language: string;
    confidence?: number;
  }>;
}
```
